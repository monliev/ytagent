## YTAgent — Rules for AI Coding Agents

> **Version:** 1.0.0
> **Last Updated:** 2025-06-26
> **Project:** YTAgent — YouTube Multi-Channel Orchestrator
> **Context:** Proxmox home lab, 2 VMs (OMV 4TB + Ubuntu Server)

## 1. Agent Identity & Purpose

YTAgent is an **autonomous AI orchestrator** that manages YouTube uploads across 10–30 music channels. The system transforms the Supervisor (user's wife) from an "Operator" into a "Strategic Supervisor" — she only approves, edits, or discards pre-generated drafts via Telegram. The Editor (user) only renders videos to OMV; the AI handles everything else.

**Key Philosophy:**
- **90% Automation, 10% Human Supervision**
- The AI must be proactive, not reactive
- The Supervisor must never need to open the dashboard for routine tasks
- The system must learn and improve over time via the feedback loop

## 2. Non-Negotiable Constraints (Hard Rules)

These rules **must never be violated** under any circumstance:

### 2.1 Upload Queue Rule
- **Uploads are strictly sequential (1-line)**. Only ONE video uploads at any given time across ALL channels.
- If Channel A (preferred time 10:00) finishes at 14:30, Channel B starts immediately — no waiting for its preferred time if the queue is clear.
- Preferred upload times are **guidelines**, not hard constraints. The queue manager uses them as scheduling hints but never blocks execution.

### 2.2 File Safety Rule
- **Video files on OMV are NEVER deleted** by the system. The user retains full ownership of master copies.
- Thumbnails generated by AI are stored alongside videos in `/nas/[channel_name]/thumbnails/`.
- All system-generated files (screenshots, thumbnails, logs) are stored in predictable subdirectories.

### 2.3 Copyright Safety Rule
- If ANY video receives a Copyright Claim, the system must **immediately STOP all upload queues** across ALL channels.
- A critical alert must be sent to the Supervisor via Telegram.
- Queue resumes only after manual Supervisor approval.
- This is a fail-safe to protect all accounts.

### 2.4 Metadata Change Rule
- The AI can **suggest** metadata changes based on performance analytics.
- The AI **must NEVER automatically apply** metadata changes to channel presets without explicit Supervisor approval.
- Every suggestion requires a [Apply] / [Ignore] decision via Telegram or Dashboard.

### 2.5 Notification Minimalism Rule
- Telegram notifications must be **minimalist**: 1–2 sentences maximum.
- Always include inline action buttons: [Approve] [Edit] [Discard].
- Never send "chatty" or verbose messages to Telegram.
- Reports (08:00 WIB) can be more detailed but must be scannable.

### 2.6 Retry Rule
- Failed uploads retry **automatically** with exponential backoff, **maximum 5 attempts total**.
- Retry delays: 5 min → 10 min → 20 min → 40 min → 60 min (exponential backoff with jitter, capped at 1 hour).
- Only after all 5 retries are exhausted should the system notify the Supervisor.
- Never require manual "Resume" clicks for transient failures (network, API timeout).
- `max_retries=5` is the single source of truth — consistent across ARCHITECTURE, CODING_STANDARDS, and this document.

## 3. Agent Responsibilities

### 3.1 As Orchestrator (Hermes Agent)

The AI agent (Hermes) is responsible for:

| Responsibility | Description |
|----------------|-------------|
| **Monitor** | Watch OMV folders for new video files (watchdog) |
| **Extract** | Capture screenshot at second 30 using ffmpeg |
| **Generate** | Create metadata drafts (title, description, tags) and thumbnail options via AI |
| **Notify** | Send minimalist Telegram notifications to Supervisor with action buttons |
| **Queue** | Manage sequential upload queue via Celery + Redis |
| **Upload** | Execute YouTube Data API v3 uploads (private mode) |
| **Collect** | Gather analytics (views, CTR, AVD) via YouTube Analytics API |
| **Analyze** | Store performance data in Qdrant, detect patterns, generate insights |
| **Report** | Send accumulative performance reports at 08:00 WIB via Telegram |
| **Alert** | Detect anomalies (CTR < 2%, views 24h < 100, AVD drop) and alert immediately |
| **Learn** | Build confidence scores for metadata, track A/B test results |
| **Protect** | Monitor copyright claims, auto-pause all queues if detected |

### 3.2 As Supervisor (User's Wife — Human)

The Supervisor's role is limited to:

| Action | When |
|--------|------|
| **Approve** | Click [Approve] in Telegram → video enters upload queue |
| **Edit** | Click [Edit] → opens dashboard to modify metadata/thumbnail/schedule |
| **Discard** | Click [Discard] → removes draft, video stays on OMV |
| **Override Schedule** | Change upload time for a specific video (doesn't affect channel default) |
| **Apply Suggestions** | Click [Apply] on AI-generated metadata improvement suggestions |
| **Resume Queue** | Click [Resume] after copyright claim auto-pause |
| **Set Channel Presets** | Define default thumbnail style, metadata templates, preferred upload time |
| **Enable Auto-Approve** | Toggle per-channel auto-approval after sufficient confidence |

### 3.3 As Editor (User — Human)

The Editor's role is minimal:

| Action | When |
|--------|------|
| **Render Videos** | Create video content, export to local machine |
| **Copy to OMV** | Move rendered files to `/nas/[channel_name]/` |
| **No system interaction required** | The system handles everything after file copy |

## 6. Communication Patterns

### 6.1 Internal Service Communication
- All internal services communicate via **FastAPI REST endpoints** or **Celery task queues**.
- No direct database access from frontend — always through API layer.
- File watcher runs as a background process, triggers API calls on detection.

### 6.4 Celery + Async Pattern (Critical)
- Celery workers run in a **prefork process pool** — they are NOT async contexts.
- **Do NOT use `asyncio.run()` inside Celery tasks** — this causes event loop conflicts and is an anti-pattern.
- All I/O inside Celery tasks must use **synchronous** code (sync SQLAlchemy, sync httpx, etc.).
- The correct pattern is to use a **synchronous database session** inside tasks:

```python
# ✅ CORRECT — Celery task uses SYNC session
from sqlalchemy import create_engine
from sqlalchemy.orm import Session

@shared_task(bind=True, max_retries=5)
def upload_video_to_youtube(self, video_id: int) -> dict:
    engine = create_engine(settings.sync_database_url)  # sync URL
    with Session(engine) as db:
        video = db.get(Video, video_id)
        result = upload_service_sync(db, video)  # sync upload logic
    return result

# ❌ WRONG — asyncio.run() inside Celery task
def upload_video_to_youtube(self, video_id: int):
    async def _do_upload():
        ...
    return asyncio.run(_do_upload())  # Will crash / create nested loop issues
```

- FastAPI routes use `AsyncSession`; Celery tasks use `Session` (sync). Both connect to the same database.
- If full async task queue is needed in the future, consider migrating to **arq** (async-native).

### 6.2 Telegram Bot Patterns
- Use **python-telegram-bot** library.
- All callbacks use **InlineKeyboardMarkup** with structured callback_data: `action:video_id:channel_id`.
- Implement conversation handlers for multi-step flows (Edit metadata, Override schedule).
- Store `chat_id` per user for targeted notifications.

### 6.3 Error Handling Patterns
- All errors are logged with structured JSON format.
- Critical errors (copyright, queue failure) trigger **immediate Telegram alert**.
- Transient errors (network, API timeout) follow **retry logic** before human notification.
- Every error includes: timestamp, service, error type, video_id (if applicable), retry count.

## 8. AI Behavior Guidelines

### 8.1 Proactivity
- The AI should **anticipate** problems, not just react to them.
- Example: If 5 consecutive videos show AVD drop at second 30, proactively suggest: "5 videos terakhir drop di detik 30 — mungkin intro terlalu panjang?"

### 8.2 Explainability
- Every AI-generated suggestion must include **why** it was made.
- Example: "CTR video ini 1.8%, di bawah rata-rata channel (3.2%). Thumbnail mungkin kurang kontras. Coba style 'high-contrast warm' untuk video berikutnya."

### 8.3 Confidence Scoring
- Every metadata draft gets a **Confidence Score (0–100%)**.
- Scores < 80% always require Supervisor review.
- Scores >= 80% can be auto-approved if auto-approve mode is ON.
- Confidence improves over time as Qdrant accumulates successful patterns.

### 8.4 Learning Loop
- Store successful metadata patterns in Qdrant.
- Compare new video performance against historical vectors.
- Identify what makes a title/description/thumbnail successful per channel.
- Feed insights back into the generation engine.

## 9. Security Rules

### 9.1 Credential Management
- Each channel has its own `client_secret.json` from GCP.
- OAuth tokens are stored **encrypted** in the database.
- Never log credentials, tokens, or secrets.
- Environment variables for all sensitive configuration.

### 9.2 GCP Quota Protection
- Monitor API usage per GCP project.
- Alert when quota usage exceeds 80%.
- Never exceed daily quota limits — queue tasks accordingly.

### 9.3 File System Security
- File watcher only monitors designated `/nas/` mount points.
- No write access to OMV except thumbnail/log subdirectories.
- Validate file types (only `.mp4`, `.mov`, `.avi`) before processing.

## 11. Dual-Channel Approval: Idempotency Guard

Video approval can arrive from **two channels simultaneously**: Telegram callback and Dashboard button. Both call the same API endpoint (`POST /videos/{id}/approve`). The system must be **idempotent** — approving an already-approved video must be a no-op, not a duplicate queue entry.

```
Telegram [Approve] ──►
                        POST /videos/{id}/approve  ──► Check status
Dashboard [Approve] ──►                                  │
                                                    status == 'staging'?
                                                    │              │
                                                   YES             NO
                                                    │              │
                                               Set 'approved'   Return 200
                                               Add to queue     (already done)
                                               Notify both UIs
```

**Implementation Rule:**
- The `/approve` endpoint must check current video status inside a **database transaction with row-level lock** (`SELECT ... FOR UPDATE`).
- If status is already `approved`, `queued`, `uploading`, or `uploaded`, return HTTP 200 with `{"status": "already_approved"}` — no duplicate queue entry.
- Telegram callback handler must edit the original message regardless of whether it was the first or duplicate approval.

**Idempotency Check (SQL):**
```sql
START TRANSACTION;
SELECT status FROM videos WHERE id = ? FOR UPDATE;
-- Only update if status = 'staging'
UPDATE videos SET status = 'approved' WHERE id = ? AND status = 'staging';
COMMIT;
```
